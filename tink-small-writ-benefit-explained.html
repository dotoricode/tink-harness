작은 Writ 방식 · Tink 적용 효과 추정

이득은 “더 똑똑해짐”보다 “덜 읽고, 덜 반복하고, 덜 놓침”입니다

Writ 전체를 가져오지 않고, Tink에 작은 버전만 넣는다고 가정했습니다: mandatory/retrievable 분리, phase별 중복 로드 방지, 검증 증거 기록, friction log, opt-in guard 승격입니다.

현재 기준 약 4.9k

Tink 하네스 본문 전체를 대략 토큰으로 환산한 값입니다. 문자 수/4 기준의 추정치입니다.

목표 60-85%

한 run에서 불필요한 하네스/규칙 로딩을 줄일 수 있는 범위입니다.

검증 2-4개

릴리스/배포 작업에서 자동으로 빠지지 않게 만들 수 있는 핵심 체크 수입니다.

운영 0 daemon

Writ식 이득 일부를 얻되 Neo4j/FastAPI/vector backend 없이 시작합니다.

쉬운 비유

지금 방식이 “작업할 때마다 도구함 설명서를 어느 정도 읽는 방식”이라면, 작은 Writ 방식은 “오늘 작업표를 먼저 쓰고, 그 작업표에 맞는 공구만 꺼내는 방식”입니다. 더 큰 차이는 작업 후입니다. 어떤 공구가 필요 없었는지, 어떤 검사가 빠졌는지, 어디서 자꾸 막혔는지를 짧게 기록해서 다음번 선택을 더 가볍게 만듭니다.

기능별 예상 이득

작은 구현 정량적 기대 사용자가 느끼는 변화 구현 비용
mandatory / retrievable 분리 항상 로드되는 규칙을 5-10개로 제한. 상황별 규칙 70% 이상 미로드 가능. 중요한 안전 규칙은 안 빠지고, 잡다한 규칙은 덜 보임. 낮음: JSON schema/selector 수정
phase별 loaded rule dedupe 긴 run에서 중복 rule injection 30-60% 감소 기대. 같은 말을 반복해서 덜 듣게 됨. 낮음: session.json 추가
verification evidence ledger 릴리스/배포에서 필수 체크 2-4개를 빠뜨릴 확률 감소. “테스트 했어?”가 아니라 “무슨 명령이 통과했어?”로 바뀜. 중간: /tink:verify 기록 확장
friction log 반복 실패를 수동 기억 대신 count로 추적. 2회 이상 반복 시 weave 후보 자동화 가능. 왜 자꾸 개선 후보가 뜨는지 근거가 명확해짐. 낮음: JSONL append
opt-in guard promotion 반복 실패가 3회 이상이면 hook guard 후보로 승격 가능. 중요한 실수는 조언이 아니라 실제 차단으로 바뀜. 중간: 승인/rollback 흐름 필요
가벼운 keyword scorer 하네스/룰 30개 이상에서 후보 선택 정확도 향상. 본문 로드 수 1-3개 유지 가능. 작업에 덜 맞는 하네스가 끼어드는 일이 줄어듦. 중간: 검색 점수화 구현

Writ 전체를 안 가져오는 이유

Writ 전체 방식
  • Neo4j Docker 컨테이너 필요
  • FastAPI daemon 필요
  • Python 3.11+와 여러 native dependency 필요
  • hook 30개 등록
  • 큰 rule corpus에는 강력하지만 설치 부담이 큼
Tink 작은 방식
  • JSON 파일 중심
  • daemon 없음
  • 기본 hook은 advisory-only
  • 반복 실패만 opt-in guard 후보
  • Claude Code와 Codex 둘 다 유지 가능

추천 로드맵

  1. 1단계: rules/index.json에 mandatory, retrievable, budget_cost, phase 필드 추가.
  2. 2단계: .tink/current/session.json에 loaded_rule_ids_by_phase 기록.
  3. 3단계: /tink:verify 결과를 verification.json 또는 ledger에 구조화.
  4. 4단계: friction.jsonl을 만들고 /tink:weave가 반복 실패를 우선순위로 정렬.
  5. 5단계: 규칙이 30-50개를 넘으면 가벼운 keyword scorer 도입.

결론

작은 Writ 방식의 가장 큰 이득은 성능 숫자 하나가 아니라 운영 습관입니다. Tink가 “많이 읽고 잘 판단하기”에서 “적게 읽고, 증거를 남기고, 반복 실패만 강화하기”로 바뀝니다. 현재 규모에서는 25-50% 절감이 현실적이고, 규칙이 늘어난 팀 사용에서는 60-85% 절감까지 기대할 수 있습니다.

기준: 현재 Tink 하네스 본문 약 19.6k chars / 2.76k words, 토큰은 chars/4로 대략 환산. Writ 수치는 저장소 문서의 retrieval/context-reduction 주장을 참고했으며, 이 문서는 Tink용 경량 구현의 추정 모델입니다.
