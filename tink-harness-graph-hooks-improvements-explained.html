Tink 개선 결과

긴 Markdown을 덜 읽고, 필요한 규칙만 고르고, 중요한 검증은 실제로 확인하게 바꿨습니다

기존에는 하네스 설명을 잘 골라 읽는 쪽에 가까웠습니다. 이제는 먼저 작업 계약서를 만들고, 그 계약서를 기준으로 rule graph가 필요한 하네스와 체크만 고릅니다. 마지막에는 /tink:verify가 계약서에 적힌 검증을 실행합니다.

이전 문제 규칙이 많아질수록 컨텍스트가 무거워짐

Markdown 하네스 본문을 많이 읽으면 정작 작업에 필요한 정보가 묻힐 수 있습니다.

새 방식 작업 계약서부터 작성

contract.json에 작업 종류, 위험, 성공 조건, 검증 명령을 먼저 적습니다.

효과 필요한 규칙만 로드

.tink/rules/index.json이 계약서를 보고 관련 하네스와 체크만 고릅니다.

한 줄 요약

Tink가 "어떤 하네스가 좋을까요?"에서 "이번 작업의 약속은 이것이고, 이 약속을 지키려면 이 규칙과 이 검증만 필요합니다"로 바뀌었습니다.

새 흐름

1

작업 계약서 생성

/tink:cast가 .tink/current/contract.json을 만듭니다. 작업 종류, 위험, 성공 조건, 금지 사항, 검증 명령이 들어갑니다.

2

Rule graph 조회

.tink/rules/index.json이 계약서를 보고 필요한 하네스, 체크, guard 후보만 고릅니다.

3

검증 실행

/tink:verify가 계약서에 적힌 명령과 수동 체크를 실행하고 결과를 기록합니다.

4

실패를 개선 후보로 보냄

검증 실패는 check_failed 신호로 남고, /tink:weave가 하네스 수정, rule graph 수정, opt-in hook guard 후보 중 어디에 반영할지 고릅니다.

바뀐 파일

새로 추가

  • commands/verify.md
  • templates/tink/rules/index.json
  • templates/tink/schemas/contract.schema.json
  • templates/tink/hooks/guards.json
  • docs/graph-contracts-and-guards.md

개선

  • cast: 계약서와 rule graph를 먼저 사용
  • weave: 하네스 수정뿐 아니라 rule/guard 후보도 분류
  • README, README.ko, docs/hooks, CHANGELOG 갱신
  • 테스트가 새 명령과 템플릿 설치를 검증

Hook은 어떻게 바뀌었나

기본 hook은 여전히 추천만 합니다. 도구를 막거나 파일을 저장하지 않습니다. 반복 실패가 쌓였을 때만 /tink:weave가 opt-in guard 후보를 제안하고, 사용자가 승인해야 실제 PreToolUse나 Stop 같은 강제 hook으로 바뀝니다.
