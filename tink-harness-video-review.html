Tink Harness 개선 후보

대상 영상은 “프롬프트 말고 하네스 — 안드레 카파시와 코덱스 워크플로우 6원칙”입니다. 공개 자막 전문은 확인하지 못했으므로, 영상 제목·게시 페이지·같은 6원칙을 설명하는 공개 원문성 자료를 함께 대조했습니다.

강점 이미 방향은 맞음 Tink는 프롬프트 묶음보다 run state, harness, memory, approval gate를 중심으로 설계되어 있습니다.
주요 공백 검증 루프 체크를 기록하는 구조는 있지만, 테스트/평가 결과가 하네스 개선으로 자동 연결되는 경로가 약합니다.
가장 큰 리스크 자기 평가 편향 Stitch는 좋은 내부 점검이지만 독립 평가자나 재현 가능한 eval harness는 아직 부족합니다.

영상/자료에서 뽑은 핵심 원칙

Map, not manual
거대한 지침서보다 작은 지도와 progressive disclosure
Tink는 skills, harnesses/index.json, 명령별 파일로 이미 꽤 잘 맞습니다. 다만 trigger metadata와 discoverability를 더 구조화할 수 있습니다.
Spec first
작업 시작 전에 성공 조건, 범위, 금지선을 명확히 함
.tink/current/plan.md, checks.md, answers.md는 좋지만 machine-readable contract가 없어서 이후 검증 자동화가 어렵습니다.
Tests prove arrival
완료 선언보다 실행 가능한 검증을 우선
현재는 사용자/에이전트가 체크를 수행합니다. 실패를 ledger와 weave 후보로 자동 연결하는 얇은 verify 단계가 필요합니다.
Architecture enforces path
프롬프트 부탁이 아니라 구조와 훅으로 경로를 제한
Hook은 advisory-only라 안전하지만, 반복 실패를 막는 opt-in deterministic guard 템플릿은 아직 없습니다.
Review catches misses
생성자와 평가자를 분리하고, 필요 시 별도 관점으로 검토
내부 Stitch는 비용이 낮은 장점이 있습니다. 고위험 작업에는 독립 review/eval harness가 추가되면 더 강합니다.
Failures feed harness
실패를 프롬프트 문장으로 때우지 않고 하네스 자산으로 승격
weave, frog, ledger.jsonl 방향은 좋습니다. 다만 post-run retrospective가 자동 습관으로 고정되어 있지 않습니다.

우선순위 개선안

P0

.tink/current/contract.json 추가

작업마다 목표, 범위, 금지선, 성공 조건, 테스트 명령, rollback 기준을 구조화해 남깁니다. 지금의 plan.md/checks.md를 대체하기보다, 자동 검증과 회고가 읽을 최소 계약으로 둡니다.

효과: specification vacuum 감소, update/weave/frog가 같은 계약을 근거로 판단 가능.

P0

/tink:verify 또는 verify harness

contract.json의 검증 항목을 실행하거나 수동 확인 결과를 기록하고, 실패를 .tink/maintenance/ledger.jsonl과 weave-queue.json에 연결합니다.

효과: “완료했습니다”를 줄이고, 실패가 다음 하네스 개선으로 이어짐.

P1

독립 평가자 패턴 추가

고위험 변경, UI, release, 보안 작업에서만 별도 review harness를 권장합니다. 기본은 내부 Stitch를 유지하고, 위험도가 높을 때만 독립 평가자로 확장합니다.

효과: 자기 평가 편향 완화. 비용은 작업 위험도에 따라 제한.

P1

Post-run compound step

cast 완료 직전에 “이번 실패/반복/검증 결과 중 재사용할 것이 있는가?”를 한 번 묻고, 저장은 기존 Reusable State Save Gate를 그대로 통과시킵니다.

효과: 프롬프트 수정 대신 하네스 개선으로 누적되는 루프 강화.

P2

Opt-in deterministic guard 템플릿

deprecated API, 금지 디렉터리, release 전 필수 테스트 같은 반복 실패를 hook 후보로 만들되 자동 설치하지 않습니다. 사용자가 승인하면 project-local hook으로만 등록합니다.

효과: “주의해” 문장을 실행 가능한 가드로 바꿈. 현재 안전 정책과도 양립 가능.

P2

Context checkpoint 정책

큰 작업에서 일정 단계마다 .tink/current/notes.md를 요약하고, 다음 행동·열린 질문·검증 상태를 갱신하도록 합니다. context anxiety와 오래 실행되는 작업의 drift를 줄입니다.

효과: 긴 작업에서 “대충 완료” 선언을 줄이고 재개 품질 향상.

현재 구현과의 매핑

잘 맞는 부분

  • templates/tink/harnesses/index.json로 전체 하네스 본문을 매번 읽지 않게 설계.
  • .tink/current/ run state로 작업 상태를 파일에 남김.
  • weave와 frog가 하네스 개선/정리를 별도 명령으로 분리.
  • Reusable State Save Gate가 프롬프트성 기억 저장을 승인 기반으로 제한.
  • Codex/Claude Code 설치 surface를 선택 가능하게 바꾼 것은 “도구를 다 켜지 말고 필요한 표면만 켠다”는 원칙과 잘 맞음.

보강할 부분

  • 계약과 검증 결과가 아직 Markdown 중심이라 자동 집계가 약함.
  • 실패가 weave 후보로 들어가는 경로가 수동적임.
  • Hook은 추천 전용이라 반복 실패를 차단하는 deterministic guard 레이어가 없음.
  • 독립 평가자/다중 관점 검토는 문서상 개념은 있지만 실행 UX가 약함.
  • context checkpoint와 compaction 전 handoff 정책이 명령 계약에 더 선명하게 들어가면 좋음.

추천 구현 순서

  1. 계약 파일부터: cast가 contract.json을 쓰게 하고, 기존 테스트는 그대로 둔 채 schema 테스트를 추가합니다.
  2. 검증 기록: 작은 verify harness를 추가해 pass/fail/manual-skipped를 ledger에 남깁니다.
  3. 회고 연결: 실패 1건 이상이면 weave-queue.json 후보를 만들되, 저장은 승인 후에만 합니다.
  4. 고위험 확장: release/security/UI 작업에만 독립 review harness를 권장합니다.
  5. Hook은 나중에: deterministic guard는 opt-in 템플릿으로 시작하고 기본 설치는 계속 off가 맞습니다.
