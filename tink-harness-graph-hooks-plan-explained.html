쉬운 설명 · graph + hooks 계획

Tink가 “큰 문서 묶음”에서 “필요한 규칙만 켜는 시스템”으로 가는 길

tink-harness-graph-hooks-review.html의 계획을 예시 중심으로 다시 설명합니다. 핵심은 Markdown을 없애자는 것이 아니라, Markdown을 얇은 안내판으로 줄이고 실제 판단·검증·강제는 구조화된 데이터와 hooks로 옮기는 것입니다.

문제 모든 규칙을 읽으면 무거움 작업이 작아도 큰 하네스 본문을 다 읽으면 컨텍스트가 빨리 찹니다.
해결 방향 필요한 규칙만 선택 작업 계약과 rule graph를 보고 지금 필요한 하네스, 체크, guard만 고릅니다.
강제 중요한 건 hook으로 막음 “주의해”라고 적는 대신, 반복 실패는 opt-in hook으로 실제 차단합니다.

한 문장으로 요약

지금 Tink는 “작업에 맞는 하네스를 고르는 도우미”에 가깝고, 다음 단계는 “작업 정보를 구조화해서 필요한 규칙만 켜고, 중요한 실수는 hook으로 막는 하네스 운영체제”에 가깝습니다.

비유로 이해하기

나쁜 방식: 매번 전체 매뉴얼 읽기

카페에서 라떼 한 잔 만들 때마다 직원에게 300쪽짜리 운영 매뉴얼 전체를 읽히는 방식입니다.

  • 시간이 오래 걸림
  • 중요한 규칙이 묻힘
  • 작업과 상관없는 내용이 판단을 흐림

좋은 방식: 상황별 체크리스트만 꺼내기

라떼 주문이면 우유/샷/온도 체크리스트만 보고, 알레르기 주문이면 알레르기 guard가 켜지는 방식입니다.

  • 컨텍스트가 작음
  • 필요한 규칙만 보임
  • 중요한 규칙은 시스템이 강제함

계획 1: contract.json

사용자가 “릴리스 준비해줘”라고 하면, Tink가 먼저 이번 작업의 계약서를 작게 만듭니다. 이 파일은 사람이 읽는 계획서가 아니라, Tink가 다음 판단에 쓰는 구조화된 입력입니다.

예시

사용자 요청: “npm 배포 전에 검증해줘.”

Tink가 만드는 계약:

  • 작업 종류: release
  • 위험도: public_publish
  • 성공 조건: tests pass, package contents checked, README commands match
  • 금지선: secrets 포함 금지, 버전 불일치 금지
  • 검증 명령: npm test, npm pack --dry-run

효과: 이후 단계가 “뭘 해야 하지?”를 추측하지 않고 같은 계약을 기준으로 움직입니다.

계획 2: Rule graph layer

Rule graph는 거창하게 시작할 필요가 없습니다. 처음에는 .tink/rules/index.json 같은 파일이면 충분합니다. 여기에는 “어떤 상황이면 어떤 하네스/체크/hook 후보를 켤지”가 작게 들어갑니다.

릴리스 작업 예시

contract.json에 task_type=release, risk=public_publish가 있으면:

  • ship 하네스를 후보로 올림
  • pre-publish-multi-agent-verify를 고위험 검증 후보로 올림
  • package secret leak check를 켬
  • README의 설치 명령과 실제 CLI help가 맞는지 확인

버그 수정 예시

task_type=bug_fix, has_repro=true이면:

  • bug-fix 하네스만 로드
  • release용 체크는 읽지 않음
  • 회귀 테스트 또는 재현 명령만 요구

효과: Markdown 본문을 검색해서 다 읽는 대신, 작은 metadata 그래프가 먼저 후보를 좁힙니다.

계획 3: /tink:verify

지금은 에이전트가 “테스트 통과했습니다”라고 말하면 사람이 믿거나 로그를 확인해야 합니다. /tink:verify는 계약에 적힌 검증을 실행하고 결과를 기록하는 단계입니다.

예시

  • contract.json에 npm test가 있음
  • /tink:verify가 실제로 npm test를 실행
  • 성공하면 ledger.jsonl에 pass 기록
  • 실패하면 어떤 하네스나 규칙을 고쳐야 할지 weave-queue.json 후보로 남김

효과: “완료 선언”보다 “검증 기록”이 중심이 됩니다.

계획 4: Opt-in enforcement hooks

모든 프로젝트에 hook을 강제로 켜면 부담이 큽니다. 대신 사용자가 승인한 프로젝트에서만 반복 실수를 hook으로 막습니다.

deprecated API 예시

어떤 프로젝트에서 에이전트가 계속 payments.v1을 사용한다고 가정합니다.

  • 처음에는 테스트 실패로 잡힘
  • weave가 “반복 실패”로 분류
  • Tink가 “payments.v1 사용을 막는 PreToolUse hook을 만들까요?”라고 제안
  • 사용자가 승인하면 이후 파일 수정 전에 hook이 감지하고 차단

릴리스 전 테스트 예시

배포 작업인데 테스트를 안 돌리고 종료하려고 하면:

  • Stop hook이 contract.json을 확인
  • 필수 검증이 비어 있으면 종료를 막고 “먼저 verify를 실행해야 함”을 알림

효과: 중요한 규칙이 프롬프트 속 조언이 아니라 실제 실행 경계가 됩니다.

계획 5: Post-run compound step

작업이 끝날 때 Tink가 한 번 묻습니다. “이번 작업에서 다음번에도 쓸 만한 실패나 규칙이 있었나?” 단, 자동 저장은 하지 않고 기존 승인 절차를 유지합니다.

예시

  • 이번 작업에서 README와 CLI help가 자주 어긋난 것을 발견
  • Tink가 “release 작업에는 CLI help와 README 명령 비교 체크를 추가할까요?”라고 제안
  • 사용자가 승인하면 rule graph의 release 노드에 체크가 추가됨
  • 다음 release 작업부터 자동으로 그 체크가 후보에 올라옴

효과: 실패가 일회성 대화로 사라지지 않고 하네스가 조금씩 똑똑해집니다.

전체 흐름 예시: npm 릴리스 검증

1

사용자 요청

“이번 버전 npm 배포 전에 검증해줘.”

2

계약 생성

Tink가 contract.json에 release, public publish, required checks를 기록합니다.

3

Rule graph 선택

release에 필요한 ship, package check, secret check만 후보로 올립니다. 버그 수정용 하네스는 읽지 않습니다.

4

검증 실행

/tink:verify가 npm test, npm pack --dry-run을 실행하고 결과를 기록합니다.

5

실패 처리

package에 private path가 들어가면 실패를 기록하고, 다음부터 막을 hook guard 후보를 제안합니다.

6

학습

사용자가 승인하면 release rule에 새 체크가 추가됩니다. 다음 릴리스부터 자동 후보가 됩니다.

왜 외부 graph DB를 바로 넣지 않나?

나중에는 좋을 수 있음

  • 규칙이 수백 개 이상으로 늘어날 때
  • 팀/조직 단위로 하네스 사용 데이터를 분석할 때
  • 여러 repo의 규칙 관계를 한 번에 다룰 때

지금은 가볍게 시작

  • Tink는 npm installer로 쉽게 설치되어야 함
  • Windows, Claude Code, Codex 양쪽에서 돌아야 함
  • 초기에는 JSON/SQLite만으로도 충분함

최종 로드맵

  1. 먼저: contract.json으로 작업 계약을 구조화합니다.
  2. 다음: /tink:verify로 검증 결과를 기록합니다.
  3. 그다음: .tink/rules/index.json으로 필요한 규칙만 고릅니다.
  4. 반복 실패가 보이면: weave가 hook guard 후보를 제안합니다.
  5. 사용자가 승인하면: opt-in hook이 실제로 차단하거나 경고합니다.
  6. 규칙이 커지면: JSON에서 SQLite, 필요하면 실제 graph DB로 승격합니다.
