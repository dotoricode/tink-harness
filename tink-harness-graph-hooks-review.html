Tink review addendum · 2026-05-25

Graph DB + Hooks 아이디어 검토

이전 리포트의 contract.json, /tink:verify, post-run compound step 계획에, “Markdown 기반 하네스는 토큰을 낭비한다. graph database와 hooks로 필요한 규칙만 적용해야 한다”는 피드백을 함께 검토했습니다.

검토 대상 주장

If a harness uses md files it's trash. We are way past using md files as they consume too many tokens and bloat the context window too much. I personally built a harness that uses an actual graph database, all skills are nodes in that database and rules only get applied as needed. I also use Claude hooks for true enforcement. Never had to use any other plugin from Claude.
타당함 Context bloat 문제 제기는 맞다 큰 Markdown을 항상 읽는 하네스는 실제로 취약합니다. 규칙은 필요한 순간에만 로드되어야 합니다.
과장됨 “md면 trash”는 너무 단정적 Claude Skills 자체도 SKILL.md를 쓰며 progressive disclosure를 전제로 합니다. 문제는 파일 형식보다 로딩 전략입니다.
접목 가치 높음 Graph + hooks는 다음 단계에 좋다 Tink의 index.json을 규칙 그래프로 발전시키고, 반복 실패는 opt-in hooks로 강제하는 방향이 적절합니다.

타당성 평가

Markdown 비판
큰 md 본문을 매번 컨텍스트에 넣으면 나쁘다.
맞습니다. 하지만 Tink는 이미 harnesses/index.json으로 요약만 먼저 보고, 필요할 때 본문을 읽는 구조입니다. 따라서 “md 사용 자체”보다 “무조건 로드”가 진짜 문제입니다.
Graph DB 아이디어
skills/rules를 node로 두고 관계와 조건으로 선택한다.
방향은 좋습니다. 다만 초기부터 실제 graph DB를 넣으면 설치 복잡도와 장애면이 커집니다. Tink에는 먼저 rules/index.json 또는 SQLite 기반 adjacency table부터 시작하는 편이 낫습니다.
Rules only as needed
상황에 맞는 규칙만 적용한다.
매우 타당합니다. 이전 계획의 contract.json이 trigger facts를 만들고, graph/rule index가 필요한 harness와 guard를 선택하면 잘 맞습니다.
Claude hooks enforcement
지침보다 lifecycle hook으로 강제한다.
공식 문서 기준으로도 강한 주장입니다. PreToolUse는 실행 전 차단, UserPromptSubmit은 프롬프트 처리 차단/컨텍스트 주입이 가능합니다. Tink의 현재 hook은 advisory-only라 다음 단계에서 opt-in guard가 필요합니다.
No other plugin needed
잘 만든 하네스면 플러그인 의존이 줄어든다.
개인 워크플로우로는 충분히 가능합니다. 다만 Tink는 배포 가능한 최소 공통 레이어라, graph DB나 hook guard를 기본 강제하면 사용자 신뢰와 설치 단순성을 해칠 수 있습니다.

이전 계획과 합친 개선안

P0

.tink/current/contract.json

목표, 범위, 성공 조건, 금지선, 검증 명령, 위험도를 구조화합니다. Graph/rule 선택의 입력 facts가 됩니다.

P0

/tink:verify 또는 verify harness

계약에 적힌 검증을 실행하고, pass/fail/manual-skip을 ledger.jsonl에 남깁니다. 실패는 weave-queue.json 후보로 연결합니다.

P1

Rule graph layer

.tink/rules/index.json 또는 SQLite 파일을 추가해 harness, guard, memory rule, check를 node로 관리합니다. 처음부터 Neo4j 같은 외부 graph DB는 피하고, repo-local file DB로 시작합니다.

예: task_type=release + surface=codex + risk=public_publish이면 ship, pre-publish-multi-agent-verify, package leak checks만 후보로 로드.

P1

Opt-in enforcement hooks

현재 advisory hook은 유지하되, 사용자가 승인한 프로젝트에 한해 PreToolUse, PostToolUse, Stop guard 템플릿을 설치합니다.

예: forbidden path write 차단, release 전 test 미실행 stop 차단, deprecated API 사용 감지.

P1

Post-run compound step

실패/반복/검증 결과를 prompt patch가 아니라 rule node, hook guard, harness edit 후보로 분류합니다. 저장은 기존 Reusable State Save Gate를 통과합니다.

P2

Markdown body diet

기존 md 하네스는 없애기보다 frontmatter와 short body로 줄이고, 긴 절차는 references/ 또는 scripts로 분리합니다. Codex/Claude 양쪽 호환성을 유지합니다.

Tink에 바로 반영할 설계 방향

채택할 것

  • 하네스 선택을 Markdown 전문 검색이 아니라 metadata/rule graph 검색으로 이동.
  • 반복 실패는 Markdown 지침 추가보다 hook guard 후보로 승격.
  • 기본 설치는 가볍게 유지하고, enforcement는 opt-in으로 분리.
  • contract.json을 graph query의 입력으로 사용.
  • 검증 실패를 weave와 rule update 후보로 자동 연결.

채택하지 않을 것

  • “md 하네스 전면 폐기”는 하지 않는 편이 낫습니다. Claude/Codex skill 생태계와 배포 단순성을 잃습니다.
  • 외부 graph DB를 기본 의존성으로 넣지 않습니다. npm installer의 설치면이 커지고 Windows/CI 문제가 늘어납니다.
  • hooks를 기본 강제하지 않습니다. 사용자는 파일 수정/명령 차단 계층을 명시적으로 이해하고 승인해야 합니다.
  • 모든 규칙을 자동 적용하지 않습니다. Tink의 핵심은 작은 유효 세트를 고르는 것이지, 숨은 자동화를 늘리는 것이 아닙니다.

추천 로드맵

  1. Phase 1: contract.json schema와 /tink:verify 추가.
  2. Phase 2: .tink/rules/index.json 추가. harness/check/guard 후보를 node-like record로 저장.
  3. Phase 3: /tink:weave가 실패를 rule update, harness edit, hook guard 후보로 분류.
  4. Phase 4: opt-in PreToolUse/Stop guard templates 추가.
  5. Phase 5: rule graph가 커질 때만 SQLite로 승격. 외부 graph DB는 enterprise/plugin variant에서 검토.

그 사람에게 답변 초안

English

I strongly agree with the core point: if a harness dumps large Markdown files into context by default, it is doing the wrong thing. The unit of design should be retrieval and enforcement, not “more instructions in the prompt.” Where I would phrase it a little differently is that Markdown itself is not the fatal flaw. Always-loaded Markdown is. A small SKILL.md or harness file can still work if it is only a thin routing layer and the real rules are loaded progressively. But your graph approach is the direction I would want a serious harness to move toward: skills/rules as addressable nodes, relationships and triggers deciding what gets loaded, and hooks enforcing the parts that must not be optional. The hooks point is especially important. A rule that matters for safety or correctness should not live only as advice in a prompt. It should become a PreToolUse, Stop, or similar guard when the project opts into that level of enforcement. That is the difference between “please remember” and “the system will not let this happen.” For Tink specifically, I would not replace everything with an external graph DB immediately, because install simplicity and cross-agent portability matter. But I would absolutely steal the architecture: keep Markdown as a small human-readable shell, add a structured rule graph/index underneath it, and promote repeated failures into opt-in Claude hooks. That gives us the best of both: low-friction distribution, smaller context, and real enforcement where it counts.

한국어

핵심 지적에는 강하게 공감합니다. 하네스가 큰 Markdown 파일을 기본으로 컨텍스트에 쏟아 넣는 구조라면 방향이 잘못된 겁니다. 하네스의 설계 단위는 “프롬프트에 지침을 더 넣기”가 아니라, 필요한 규칙만 검색하고 강제하는 구조여야 합니다. 다만 저는 Markdown 자체가 문제라고까지는 말하지 않을 것 같습니다. 진짜 문제는 항상 로드되는 Markdown입니다. 작은 SKILL.md나 하네스 파일은 얇은 라우팅 계층으로 남겨두고, 실제 규칙은 progressive disclosure로 필요한 순간에만 불러오면 여전히 유효합니다. 하지만 말씀하신 graph 방식은 진지한 하네스가 가야 할 방향에 가깝습니다. skill과 rule을 노드로 두고, 관계와 trigger가 무엇을 로드할지 결정하게 하는 방식이 훨씬 낫습니다. hooks 이야기는 특히 중요합니다. 안전이나 정확성에 중요한 규칙은 프롬프트 안의 조언으로만 남아 있으면 안 됩니다. 프로젝트가 동의한 경우에는 PreToolUse, Stop 같은 hook guard로 승격되어야 합니다. 이게 “기억해줘”와 “시스템이 실제로 막는다”의 차이입니다. Tink에 바로 외부 graph DB를 기본 의존성으로 넣지는 않는 편이 좋다고 봅니다. 설치 단순성, Windows 호환성, Claude/Codex 양쪽 이식성이 중요하기 때문입니다. 하지만 아키텍처 아이디어는 충분히 가져올 가치가 있습니다. Markdown은 사람이 읽는 얇은 shell로 줄이고, 아래에는 구조화된 rule graph/index를 두며, 반복 실패는 opt-in Claude hooks로 승격하는 방식입니다. 그러면 배포는 가볍게 유지하면서도 컨텍스트는 줄이고, 정말 중요한 규칙은 강제할 수 있습니다.
