GitHub review · infinri/Writ · 2026-05-26

Writ은 “규칙 검색 엔진 + 강제 게이트”에 가까운 Claude Code 하네스입니다

Writ은 큰 rulebook을 그대로 프롬프트에 넣지 않고, BM25·vector·graph traversal을 섞어 필요한 규칙만 뽑습니다. 동시에 hook과 세션 상태 머신으로 계획 승인, 테스트 승인, 검증 증거를 강제합니다.

핵심 1 빠른 librarian

작업 문맥을 보고 관련 규칙만 검색합니다. README 기준 276-rule corpus에서 p95 0.590ms를 주장합니다.

핵심 2 process keeper

Work mode에서 계획 승인과 테스트 준비가 끝나기 전 위험한 쓰기를 hook으로 막습니다.

주의점 기본 설치가 무겁습니다

Python 3.11, Docker, Neo4j, FastAPI daemon, Tantivy, hnswlib, ONNX Runtime이 들어갑니다.

한 줄 평가

Writ은 Tink가 이번에 추가한 contract.json + rules/index.json 방향의 훨씬 무거운 완성형입니다. 아이디어는 매우 참고할 만하지만, Tink의 기본값으로 그대로 들여오면 설치 복잡도와 hook 부담이 Tink의 장점을 해칠 가능성이 큽니다.

Writ 구조

Layer 1

Knowledge layer

FastAPI 서비스가 Neo4j graph와 인메모리 인덱스를 들고 있다가 /query 요청에 필요한 규칙만 반환합니다.

Layer 2

Retrieval pipeline

Domain filter → BM25 → vector ANN → graph traversal → ranking/context budget 순서로 규칙을 줄입니다.

Layer 3

Enforcement layer

Claude Code hook 30개가 세션 상태, mode, phase, gate, 검증 증거를 읽고 write/edit/stop 시점에 차단하거나 규칙을 주입합니다.

Layer 4

Feedback loop

마찰 로그와 feedback signal로 규칙 효과, gate denial, quality judge override, 사용 빈도를 분석합니다.

Tink와 비교

항목 Writ Tink에 맞는 해석
목표 큰 rule corpus에서 관련 규칙을 검색하고 workflow를 강제 Tink는 하네스 선택과 실행 상태 관리가 중심. 강제는 opt-in으로 제한하는 편이 맞음.
저장소 Neo4j graph + Markdown bible export/import 기본은 JSON rule graph 유지. 규칙 수가 커질 때만 SQLite/daemon 선택지를 검토.
검색 BM25 + vector + graph traversal + ranking 1차로 BM25/keyword score만 도입해도 충분. vector는 Tink 기본값에는 과함.
강제 계획·테스트·검증을 hook으로 강하게 차단 Tink는 기본 advisory, 반복 실패에 한해 opt-in guard 후보로 승격.
세션 상태 session cache가 loaded rules, budget, mode, phase, gates, evidence를 관리 Tink의 .tink/current/에 lightweight session metadata를 추가할 가치 있음.

접목 가능성

바로 가져올 만함
  • mandatory vs retrieved split: 항상 필요한 안전 규칙과 상황별 규칙을 분리.
  • context budget mode: tiny/standard/full처럼 규칙 출력량을 예산에 맞춤.
  • loaded rule dedupe: 이미 이번 phase에 쓴 rule은 다시 로드하지 않음.
  • verification evidence: 완료 주장 전에 검증 증거를 구조화해 확인.
  • friction log: gate denial, skipped verification, repeated failure를 JSONL로 남김.
기본값으로는 과함
  • Neo4j + Docker 필수 설치
  • FastAPI daemon 상시 실행
  • 30개 hook 전면 등록
  • ONNX/vector retrieval 기본 포함
  • global Claude config patching

Tink 적용 제안

우선순위 제안 이유 구현 형태
P0 Rule graph에 mandatory, retrievable, budget_cost 필드 추가 항상 필요한 규칙과 상황별 규칙을 분리하면 Markdown 로딩이 더 줄어듦. templates/tink/rules/index.json schema 확장
P0 /tink:verify가 evidence ledger를 더 명확히 기록 Writ의 verification evidence 개념은 Tink의 contract-first 흐름과 잘 맞음. .tink/current/verification.json 또는 ledger entry 확장
P1 Phase별 loaded rule dedupe 같은 실행 중 같은 규칙을 계속 다시 보여주는 낭비를 줄임. .tink/current/session.json에 loaded_rule_ids_by_phase
P1 Friction log와 /tink:weave 연결 실패, 차단, 검증 누락을 근거로 weave 후보를 자동 정렬 가능. .tink/maintenance/friction.jsonl + weave queue
P2 가벼운 BM25/keyword scorer 규칙이 50개 이상으로 늘면 단순 trigger match보다 점수화가 유리. Node 내장 또는 작은 JS dependency. daemon 없음.
P3 선택형 advanced retrieval backend 사용자가 큰 rule corpus를 운영할 때만 Writ식 graph/RAG backend가 의미 있음. 별도 plugin/skill 또는 optional adapter

권장 결론

  1. Writ의 방향성은 맞습니다. 특히 “규칙을 프롬프트에 다 넣지 말고 검색하라”는 점은 Tink의 graph/hooks 개선 방향과 일치합니다.
  2. 하지만 Tink는 installer가 가볍고 Claude/Codex 양쪽에서 돌아가는 것이 장점입니다. Writ의 Neo4j/FastAPI/ONNX stack은 기본값으로 넣지 않는 편이 낫습니다.
  3. 가장 좋은 접목은 “Writ의 개념을 Tink식으로 작게 구현”하는 것입니다: mandatory split, phase별 dedupe, evidence ledger, friction log, opt-in guard promotion.
  4. 규칙 수가 수백 개 이상으로 커지는 사용자를 위해서만 optional advanced backend를 검토하면 됩니다.
