Tink 컨텍스트 엔지니어링
현재 원리와 고도화 효과

Tink의 현재 컨텍스트 전략은 “많이 읽기”가 아니라 “작게 고르고, 왜 골랐는지 남기고, 검증과 연결하기”입니다. 다음 고도화는 여기에 역할, 비용, 재사용 신호를 붙여 컨텍스트 선택 품질을 측정 가능한 상태로 만드는 작업입니다.

기준 버전: v1.2.2 대상: Claude Code + Codex OS: macOS + Windows 새 public command 없음

1. 현재 컨텍스트 동작 원리

현재 Tink는 작업을 시작할 때 관련 파일을 무작정 많이 넣지 않습니다. 먼저 작업 계약을 만들고, repo signal과 Context Graph Lite를 이용해 작은 후보만 고른 뒤, 포함/제외 이유를 run state에 남깁니다.

작업 계약 contract.json에 작업 종류, 완료 조건, 금지 행동, 검증 방법을 적습니다.
변경 경로 수정된 path가 어떤 command, schema, test, docs와 연결되는지 봅니다.
Repo Signal context_graph_rule, sync group, verification hint로 관련성을 설명합니다.
Context Map included, excluded, signals, external_context를 기록합니다.
검증 연결 선택한 context가 어떤 check로 이어지는지 verification.json에 남깁니다.

2. 현재 효율성: 이미 좋아진 부분

작은 context pack

예시 fixture 기준으로 실제 포함 파일은 6개, 제외 항목은 2개입니다. 전체 repo를 훑는 방식보다 작은 출발점입니다.

예: command copy, Codex skill, schema, test만 우선 포함

제외 이유가 보임

excluded-context.md와 context-map.json.excluded가 왜 제외했는지 설명합니다.

예: 외부 research는 이 slice에 불필요, installer는 이미 generic copy

검증과 연결됨

context signal이 verification hint로 이어져 “왜 이 테스트가 필요한지”를 설명합니다.

예: command 변경 → sync check → npm test

3. 현재 한계

한계 현재 상태 문제
역할 구분 부족 kind, reason, confidence는 있음 source of truth인지, 예시인지, 검증 대상인지가 필드로 분리되어 있지 않음
비용 측정 부족 작게 고르라는 규칙은 있음 어떤 context가 토큰/시간을 얼마나 쓰는지 비교하기 어려움
재사용 신호 부족 run마다 context를 기록함 자주 필요한 context와 자주 불필요했던 context를 자동으로 구분하기 어려움
효과 측정 부족 검증 결과는 남김 context 선택이 성공률, 재작업률, 리뷰 시간에 얼마나 기여했는지 추적이 약함

4. 제안 작업: Context Budget Ledger + Role Tagging

추가할 필드

  • role: primary, supporting, verification_target, external_evidence, example_only, avoid_next_time
  • cost: low, medium, high 또는 rough token band
  • reuse_signal: always, often, rare, avoid_next_time
  • verification_link: 연결되는 check 또는 evidence ref
  • staleness: fresh, aging, stale, unknown

동작 방식

  • cast가 context 후보를 고를 때 역할과 비용을 함께 기록합니다.
  • verify가 실제로 쓰인 context와 쓰이지 않은 context를 evidence로 남깁니다.
  • 다음 run에서 자주 맞았던 context는 우선하고, 반복적으로 불필요했던 context는 낮춥니다.
  • 새 command 없이 기존 context-map.json, context-diff.json, docs/test만 확장합니다.

5. 예상 개선 효과

중요: 아래 수치는 아직 production telemetry가 없는 상태에서 산정한 보수적 추정치입니다. 기준은 현재 fixture 구조, 최근 PR 작업 패턴, 일반적인 agent context selection 비용을 조합한 것입니다. 실제 수치는 ledger가 쌓인 뒤 보정해야 합니다.

불필요 context 포함률 감소 20-35%

역할과 reuse signal이 생기면 example_only, stale, avoid_next_time 항목을 더 빨리 제외할 수 있습니다.

초기 context pack 크기 감소 10-25%

현재도 작은 편이라 극적인 감소보다는, 반복 작업에서 안정적으로 줄어드는 효과가 큽니다.

리뷰자가 근거 찾는 시간 감소 30-50%

role과 verification_link가 있으면 “왜 이 파일을 봤는지”를 diff에서 다시 추리하지 않아도 됩니다.

검증 누락 탐지율 개선 15-30%

verification_target role이 있는데 check가 없으면 바로 누락으로 볼 수 있습니다.

반복 작업 context 재사용 정확도 25-45%

update, release, cast-command-sync처럼 반복되는 작업에서 효과가 먼저 나타납니다.

재작업 가능성 감소 10-20%

context 자체가 검증과 연결되면 “파일은 봤지만 check를 빼먹은” 상황이 줄어듭니다.

6. 정량 산정 방식

지표 현재 기준 고도화 후 측정 방식
Context precision 포함된 context 중 실제 검증/수정에 연결된 비율을 수동 추정 role + verification_link가 있는 항목 비율로 측정
Context waste excluded 항목은 보이지만, 포함 후 쓸모 없었던 항목은 약함 reuse_signal: avoid_next_time와 context-diff를 누적
Review trace time 리뷰자가 context-map과 diff를 함께 읽어야 함 각 included entry가 역할/검증 링크를 바로 제공
Repeat task quality Context Graph Lite rule 중심 작업 타입별 ledger로 자주 맞는 context를 우선 선택
