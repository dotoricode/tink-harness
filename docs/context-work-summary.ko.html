Tink 컨텍스트 엔지니어링 작업 요약

이번 연속 작업은 Tink가 “어떤 context를 왜 읽었는지”, “무엇을 제외했는지”, “어떤 검증과 연결되는지”를 더 작고 명확하게 남기도록 만드는 흐름이었습니다. 최종적으로 여섯 가지 컨텍스트 효율 지표는 fixture와 대표 run-history rollup 기준으로 모두 90% 이상을 통과했습니다.

현재 상태

main 최신open PR 없음npm 1.4.0 유지

문서, fixture, 테스트 기준은 merge 완료됐고 이번 변경들은 publish 대상이 아닌 내부 검증/문서 개선입니다.

검증

npm testgit diff --checknpm pack --dry-run --json

PR CI와 merge 후 로컬 검증을 모두 통과했습니다.

제외한 것

Sentry 제외tink index 없음release evidence bundling 제외

사용자 영역을 침범할 수 있는 자동 수집이나 공개 명령 추가는 하지 않았습니다.

작업 흐름

PR #8
Context Budget Ledger

context entry에 role, cost, reuse_signal, verification_link 같은 필드를 붙여 “왜 읽고 왜 뺐는지”를 기록할 수 있게 했습니다.

PR #9
Context Metrics Evaluator

여섯 지표를 fixture에서 계산해 90% 이상인지 테스트로 확인하는 기준을 만들었습니다.

PR #10
Context Metrics Artifact

설치되는 schema와 cast 지침이 `.tink/current/context-metrics-evaluation.json`을 알도록 연결했고, v1.4.0으로 publish/release했습니다.

PR #11
Context Run History Rollup

단일 run이 아니라 여러 대표 run의 평균과 최저점을 계산해 반복 작업에서도 90% 이상인지 확인했습니다.

PR #12
Context Threshold Status

current-run 점수, rollup 평균, rollup 최저점을 한곳에서 볼 수 있는 상태판 fixture와 문서를 추가했습니다.

다음 작업 방향

다음 단계는 실제 `.tink/runs/*` 기록이 충분히 쌓였을 때 같은 계산식으로 rollup할 수 있도록 “실제 run 기록 산출 규칙”을 문서와 fixture로 먼저 고정하는 작업입니다. 이번 단계에서는 자동 수집, watcher, hidden cache, 새 public command를 만들지 않고, 사용자가 승인한 run 기록만 대상으로 삼는 경계를 명확히 합니다.

중요: 실제 production-wide 90%라고 주장하려면 repository fixture가 아니라 누적된 실제 run 기록이 필요합니다. 그래서 다음 작업은 바로 자동화를 만들기보다, 어떤 기록을 안전하게 쓸 수 있는지부터 정합니다.
