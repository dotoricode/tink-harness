Tink release note

v1.2.0은 Tink를 “실행 전 선택”에서 “근거가 남는 실행”으로 끌어올린 패치입니다.

이번 작업의 핵심은 Claude Code와 Codex를 같은 기준으로 지원하면서, 큰 작업을 시작할 때 어떤 context를 쓰고 뺐는지, 무엇으로 검증했는지, 외부 정보를 얼마나 신뢰할 수 있는지 남기는 것입니다.

npm feature release: v1.2.0 latest display patch: v1.2.1 verified: npm test 31 tests surfaces: Claude Code + Codex platforms: macOS + Windows

한눈에 보기

Before

작업은 진행됐지만 근거가 흩어졌습니다.

  • Codex 스킬 목록에 넓은 tink 스킬이 보여서 action이 흐릿했습니다.
  • 큰 작업에서 어떤 파일과 외부 정보를 기준으로 삼았는지 추적하기 어려웠습니다.
  • 검증 결과가 pass/fail/blocked/skipped로 구조화되어 있지 않았습니다.
  • Figma, GitHub, docs 같은 외부 context를 안전하게 다루는 기준이 부족했습니다.
After

이제 선택, 실행, 검증의 흔적이 남습니다.

  • Codex는 $tink:cast, $tink:verify 같은 action skill 중심으로 정리됐습니다.
  • context-pack.md, context-map.json, excluded-context.md로 context 근거를 남깁니다.
  • verification.json이 검증 evidence를 구조화합니다.
  • MCP Safe Profile이 외부 context의 source, confidence, sensitivity, verification hint를 기록합니다.

개선된 점

Codex surface

스킬 목록이 action 중심으로 정리됨

넓은 tink 스킬을 없애고, 사용자가 바로 고를 수 있는 $tink:cast, $tink:verify, $tink:list 같은 alias를 전면에 둡니다. 공유 규칙은 보이지 않는 tink-core/RULES.md로 이동했습니다.

Context artifacts

작업 context가 눈에 보이게 됨

context-pack.md는 사람이 읽는 요약, context-map.json은 기계가 읽는 근거, excluded-context.md는 의도적으로 뺀 것을 기록합니다.

Repo Signals

새 명령 없이 관련 검증을 고름

새 tink index 명령을 만들지 않고, fixture 기반 repo signal로 테스트, 스키마, 동기화 파일, verification hint를 선택하는 근거를 남깁니다.

Verify Runner

검증 결과가 구조화됨

.tink/current/verification.json이 pass, fail, blocked, skipped를 구분하고, final report, notes summary, maintenance signal 흐름까지 테스트 fixture로 고정합니다.

MCP Safe Profile

외부 context를 안전하게 다룸

Figma, GitHub, official docs, dashboards, attachments 같은 외부 source를 source_ref, confidence, sensitivity, verification_hint와 함께 기록합니다. Sentry는 이번 active source set에서 제외했습니다.

Compatibility

Claude Code + Codex, macOS + Windows 기준

새 작업은 두 agent surface와 두 OS 계열을 함께 고려해야 한다는 compatibility policy를 문서화했습니다. 명령도 npm, node, python 중심의 portable check를 선호합니다.

사용할 때 무엇이 좋아지나

큰 작업을 시작할 때

Tink가 어떤 파일, 스키마, 문서, 외부 source를 기준으로 삼았는지 먼저 남기므로, 중간에 방향이 흔들려도 “처음에 무엇을 믿고 시작했는지” 되짚을 수 있습니다.

릴리스나 PR을 준비할 때

검증 evidence가 명령 출력 하나에 묻히지 않고, checked items, remaining work, next action으로 남습니다. PR 설명에도 문제, 해결, 검증, 참고 흐름을 그대로 옮기기 쉬워졌습니다.

검증 근거

로컬 소스 검증

  • npm test 통과: 31 tests OK
  • Claude command 3-copy sync 확인
  • Codex visible action skill set 확인
  • Sentry가 active external context set에 없음을 테스트로 보증

published package 검증

  • tink-harness@1.2.0 npm tarball 존재
  • v1.2.0 tarball에 핵심 docs, schemas, commands, Codex skills 포함
  • legacy templates/codex/skills/tink/SKILL.md는 tarball에 없음
  • v1.2.1은 README/GitHub Release 표시 정합성 패치로 분리

다음 계획 제안

Slice 1

릴리스 후 정리

로컬에 남은 .agents/, AGENTS.md, .tink/maintenance/ledger.jsonl을 커밋할지, ignore할지, 로컬 전용으로 둘지 결정합니다.

Slice 2

업데이트 사용자 시나리오 검증

기존 v1.1.x 또는 v1.2.0 설치 사용자가 npx tink-harness@latest update를 실행했을 때 Codex legacy skill cleanup과 새 schemas가 기대대로 들어오는지 smoke test를 추가합니다.

Slice 3

작업 상태 UX 개선

.tink/current/verification.json, context-map.json, excluded-context.md를 사람이 더 쉽게 읽도록 짧은 summary command 또는 README guide를 설계합니다. 새 command를 만들지는 않고 기존 list나 docs에서 시작합니다.

Slice 4

Phase 5 후보: update confidence

설치/update/publish 이후 실제 사용자가 가장 먼저 만나는 흐름을 단단하게 만듭니다. 목표는 “설치 후 어떤 명령을 쓰면 되는지, 업데이트로 무엇이 바뀌는지, 문제가 생기면 어디를 보면 되는지”를 줄이는 것입니다.

추천 순서는 Slice 1 → Slice 2입니다. 지금은 릴리스 직후라 기능을 더 늘리기보다, 설치된 사용자가 안전하게 최신 구조로 넘어오는지 확인하는 쪽이 효과가 큽니다.
