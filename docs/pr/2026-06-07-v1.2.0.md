# PR draft: v1.2.0 Tink context, verification, and Codex surface

## 문제

이번 작업 전에는 Tink가 Claude Code와 Codex를 함께 지원한다고 말하면서도, 실제 사용 표면과 실행 기록은 아직 충분히 정리되어 있지 않았다.

- Codex 스킬 목록에는 넓은 `tink` 스킬과 action alias가 함께 노출될 수 있어 `Tink: Tink`처럼 어색한 항목이 보였다.
- 큰 작업을 시작할 때 어떤 repo context, 외부 context, 제외한 context를 기준으로 판단했는지 추적하기 어려웠다.
- `/tink:verify`가 해야 할 일을 문서로 말하긴 했지만, pass/fail/blocked/skipped evidence shape와 maintenance signal 흐름이 충분히 구조화되어 있지 않았다.
- Figma, GitHub, official docs, dashboard, attachment 같은 외부 context를 가져올 때 무엇을 포함하고 무엇을 제외해야 하는지 안전 기준이 필요했다.
- 새 작업은 Claude Code와 Codex, macOS와 Windows를 모두 고려해야 한다는 기준을 명시적으로 남길 필요가 있었다.

## 해결

- Codex 설치 표면을 action skill 중심으로 정리했다.
  - `$tink:cast`, `$tink:verify`, `$tink:list`, `$tink:frog`, `$tink:weave`, `$tink:setup`, `$tink:update`가 visible skill로 남는다.
  - 공유 규칙은 `templates/codex/skills/tink-core/RULES.md`로 이동했다.
  - 기존 Codex install에 남아 있을 수 있는 legacy `skills/tink/SKILL.md`는 Tink legacy skill로 확인될 때 설치/update 중 정리한다.
- `/tink:cast`에 context artifact 흐름을 추가했다.
  - `context-pack.md`: 사람이 읽는 context 요약.
  - `context-map.json`: included/excluded/signals/external_context 구조.
  - `excluded-context.md`: 의도적으로 제외한 파일, source, claim 기록.
- Repo Signal을 fixture 기반으로 추가했다.
  - 새 public command인 `tink index`는 만들지 않았다.
  - 신호는 tests, schemas, sync partners, verification hints를 선택하는 보조 evidence로만 사용한다.
- Verify Runner를 구조화했다.
  - `.tink/current/verification.json` 스키마를 추가했다.
  - pass/fail/blocked/skipped, final report, notes summary, maintenance output을 테스트 fixture로 고정했다.
  - Claude Code `/tink:verify`와 Codex `$tink:verify`가 같은 runner model을 쓰도록 했다.
- MCP Safe Profile을 추가했다.
  - 외부 context는 source, source_ref, included, excluded, reason, confidence, sensitivity, verification_hint로 기록한다.
  - Sentry는 이번 active source set에서 제외했다.
  - external context가 완료 판정에 영향을 주면 `contract.verification.manual_checks[]`와 verify evidence에 연결한다.
- README, 한국어 README, CHANGELOG, VERSIONING을 v1.2.0 기준으로 정리했다.

## 참고

- 이번 버전은 npm latest `1.1.1` 이후 publish될 `1.2.0` 릴리스로 준비했다.
- `ROADMAP.md`는 프로젝트 규칙에 따라 커밋 대상에 포함하지 않는다.
- `.tink/` run records는 작업 기록이지만 package publish 대상은 아니다.
- publish 전에는 `npm pack --dry-run --json` 결과와 npm 인증 상태를 한 번 더 확인해야 한다.
