# Matt Pocock의 Agentic Engineering Workflow 분석

출처: [Matt Pocock's Agentic Engineering Workflow (just copy him)](https://youtu.be/nQwJVHCtDDY?si=eT7kNfKsb8osVOyv)
채널/진행: David Ondrej
영상 길이: 약 62분
분석 기준: YouTube 영어 자동 자막 기반. 자동 자막 특성상 일부 용어와 고유명사는 오인식 가능성이 있다.

## 한 줄 요약

영상의 핵심은 "더 좋은 모델을 기다리기보다, 모델이 일하는 환경인 하네스와 코드베이스를 개선하라"는 주장이다. Matt Pocock은 AI가 전술적 코딩을 상당 부분 대체했기 때문에, 개발자의 차별점은 전략적 판단, 작업 분해, 테스트, 코드 구조, 리뷰 시스템, 그리고 반복 가능한 agent workflow를 설계하는 능력에 있다고 본다.

## 핵심 주장

### 1. 모델보다 하네스에 더 많은 통제권이 있다

Matt은 사람들이 새 모델 성능에 과하게 집중한다고 본다. 모델 성능은 중요하지만, 사용자가 직접 개선할 수 있는 영역은 모델 자체보다 하네스다.

여기서 하네스는 다음을 포함한다.

- 프롬프트와 스킬
- 에이전트가 실행되는 샌드박스/환경
- 코드베이스 구조
- 테스트와 리뷰 절차
- 작업 큐, GitHub Actions, PR 자동화
- 에이전트가 참고할 수 있는 로컬 문서와 상태

좋은 하네스는 더 낮은 비용의 모델도 충분히 일하게 만든다. 반대로 코드베이스가 복잡하고 테스트가 약하면, 더 비싼 모델이 더 많은 토큰을 쓰며 헤매게 된다.

### 2. AI는 전술적 프로그래밍을 먹었고, 사람은 전략적 프로그래밍으로 이동해야 한다

영상 초반에 John Ousterhout의 tactical programming / strategic programming 구분이 나온다.

- 전술적 프로그래밍: 당장의 코드 작성, 문법, 버그 수정, 커밋 생성
- 전략적 프로그래밍: 코드베이스가 어떻게 생겨야 하는지, 어떤 인터페이스와 테스트가 필요한지, 장기적으로 변경 속도를 어떻게 높일지 판단

Matt의 관점에서는 AI가 전술적 작업을 매우 싸고 빠르게 수행한다. 따라서 사람의 가치는 "코드를 직접 얼마나 많이 쓰는가"보다 "무엇을 만들고, 어떤 단위로 나누고, 어떤 기준으로 검증할 것인가"에 더 강하게 걸린다.

### 3. 개발자의 실력은 AI 출력의 상한이다

AI는 시니어 개발자를 훨씬 강하게 만들지만, 기초가 약한 사람에게 같은 폭의 효과를 주지는 않는다고 말한다. 이유는 AI에 좋은 맥락을 주고, 결과를 판단하고, 구조를 설계하는 능력이 결국 사람 쪽에 있기 때문이다.

이 관점에서 배워야 할 것은 단순 문법보다 다음에 가깝다.

- Git으로 변경을 안전하게 관리하는 법
- 에러를 읽고 디버깅하는 법
- 테스트로 신뢰를 만드는 법
- 소프트웨어가 실제로 배포되는 흐름
- 코드베이스를 변경하기 쉽게 설계하는 법
- 제품 방향과 사용자 문제를 판단하는 법

### 4. 좋은 스킬은 "지식"보다 "절차"를 재사용 가능하게 만든다

Matt은 에이전트 스킬을 크게 두 종류로 나눈다.

- ability: 모델이 필요할 때 자동으로 불러오는 능력 또는 기준
- procedure: 사용자가 의도적으로 호출해서 특정 절차를 수행하게 하는 워크플로우

그는 개인적으로 procedure를 선호한다. 이유는 사람이 운전대를 잡고, 언제 어떤 절차를 쓸지 판단하는 편이 더 안전하고 예측 가능하기 때문이다.

예시로 언급한 `grill me` 스킬은 구현 전 모델을 비판적 인터뷰어로 바꾸는 짧은 절차다. 사용자의 아이디어를 바로 구현하지 않고, 중요한 결정과 모호한 부분을 질문하게 만든다. 이는 plan mode를 대체하거나 보완하는 방식으로 쓰인다.

### 5. 스킬은 무조건 많이 넣을수록 좋은 것이 아니다

스킬이나 플러그인, MCP 서버가 많아지면 각 설명이 컨텍스트에 새어 들어간다. 그 결과 모델의 기본 추론 공간이 오염되거나, 불필요한 지시가 쌓일 수 있다.

영상 마지막의 실천 조언은 꽤 급진적이다.

- 모든 스킬, 플러그인, MCP 서버, `CLAUDE.md`, `AGENTS.md`를 제거한 빈 상태에서 시작해 본다.
- 기본 에이전트가 어떻게 행동하는지 관찰한다.
- 실제로 아쉬운 지점이 생길 때만 절차형 스킬을 하나씩 되돌린다.
- 각 스킬은 사용자가 커스터마이즈하고 실험할 수 있어야 한다.

이는 Tink의 하네스 설계에도 직접 연결된다. 기본 컨텍스트를 비대하게 만드는 대신, 실패나 반복 작업이 관찰될 때 하네스를 추가하는 방식이 더 건강하다.

## Teach Skill 데모에서 나온 설계 원칙

영상 중반에는 Matt이 만든 `teach` 스킬을 시연한다. 사용자는 "나는 vibe coder이고 더 나은 소프트웨어를 만들기 위해 지식 격차를 메우고 싶다"고 요청한다. 스킬은 바로 강의를 생성하지 않고 먼저 사용자의 미션을 묻는다.

### 중요한 점

- 학습은 정보 주입이 아니라, 사용자가 세계 안에서 무엇을 할 수 있게 되는지에 맞춰야 한다.
- 스킬은 `mission.md`, learning record, reference cheat sheet, lesson HTML 같은 로컬 상태를 저장한다.
- 상태를 저장하는 스킬은 좋은 교사처럼 이전 학습, 목표, 현재 수준을 기억한다.
- 첫 수업은 Git처럼 "안전하게 변경하는 능력"에서 시작한다.
- 퀴즈와 실습을 통해 기억 강도를 높인다.
- 터미널 텍스트만 고집하지 않고 HTML 산출물로 학습 경험을 개선한다.

이 부분은 Tink의 memory / harness / verification 구조와 잘 맞는다. 단발성 프롬프트보다, 사용자의 목표와 과거 선택을 로컬 기록으로 남기는 상태형 하네스가 더 큰 효과를 낼 수 있다.

## Matt의 실제 Agentic Engineering Setup

영상에서 설명한 작업 방식은 다음과 같다.

- 로컬에서는 Claude Code를 주로 계획과 일부 구현에 사용한다.
- 많은 작업은 AFK 방식으로 처리한다.
- Sand Castle이라는 샌드박스 기반 도구를 사용해 에이전트를 격리 실행한다.
- Docker, Podman, Vercel sandboxes 같은 환경을 활용해 병렬 에이전트를 띄운다.
- GitHub Actions에서 PR 리뷰, 구현, 탐색 작업을 자동 실행한다.
- 결과는 PR이나 커밋으로 돌아오고, 사람은 리뷰와 판단에 집중한다.

핵심은 "사람이 계속 옆에서 승인 버튼을 누르는 방식"을 줄이고, 명확히 스코프된 작업을 에이전트에게 맡긴 뒤 결과를 검토하는 것이다.

## Loop보다 Queue

영상 후반의 중요한 구분은 loop와 queue다.

Matt은 무한히 돌아가는 agentic loop에 회의적이다. 실제 개발은 보통 무한 루프가 아니라 작업 큐에 가깝다고 본다.

- 이슈가 큐에 들어온다.
- 사람이 또는 자동화가 triage한다.
- 에이전트가 탐색하거나 구현한다.
- 결과가 PR이나 리뷰 대상으로 돌아온다.
- 검증 후 병합되면 큐에서 빠진다.

이 관점은 Tink에도 유용하다. "계속 돌리는 루프"보다 "작업 단위, 검증 기준, human checkpoint, 완료 조건이 있는 큐"가 더 관리 가능하다.

## Human-in-the-loop 체크포인트

Matt은 사람의 개입을 가능한 한 오른쪽으로 밀 수는 있지만, 완전히 없애는 데는 신중하다.

리뷰에는 두 가지 가치가 있다.

- 위험한 변경이 production에 들어가는 것을 막는다.
- 하네스가 제대로 작동하는지 관찰하고 개선할 피드백을 얻는다.

따라서 리뷰는 단순히 코드만 보는 행위가 아니다. 에이전트 시스템 자체를 리뷰하는 관측 지점이다. 자동 병합 범위를 넓힐 수는 있지만, 에이전트가 "이건 리뷰하지 않아도 된다"고 판단하는 기준 자체도 계속 샘플링하고 검증해야 한다.

## 제품과 비즈니스에 대한 관점

AI 시대에도 사업 만들기의 기본은 크게 바뀌지 않았다고 말한다.

- 고객과 직접 이야기한다.
- 실제 문제가 무엇인지 찾는다.
- 필요한 것을 프로토타입으로 만든다.
- 기능을 더하는 것보다 줄이는 질문을 한다.
- 제품 비전과 우선순위는 사람이 책임진다.

AI는 구현 속도를 크게 높이지만, 무엇을 만들어야 하는지와 왜 만들어야 하는지는 여전히 사람이 판단해야 한다.

## Tink Harness 관점의 적용 포인트

### 1. 하네스는 모델 보정 장치가 아니라 작업 환경이다

Tink 문서와 스킬은 "모델이 부족하니 지시문을 덧대는 패치"가 아니라, 에이전트가 일하기 쉬운 환경을 만드는 방향이어야 한다.

구체적으로는 다음이 중요하다.

- 입력 조건
- 작업 스코프
- 확인해야 할 파일
- 허용되는 변경 범위
- 검증 명령
- 완료 조건
- 실패 시 기록 방식

### 2. 하네스 추가 기준은 관찰된 반복 문제여야 한다

스킬과 하네스가 많아지면 컨텍스트 비용과 선택 복잡도가 증가한다. 따라서 새 하네스는 "있으면 좋겠다"보다 "반복적으로 같은 문제가 발생했다"는 증거가 있을 때 추가하는 편이 좋다.

이 기준은 현재 Tink의 lifecycle signal, memory evidence, update confidence 문서들과도 잘 맞는다.

### 3. Procedure-first 설계가 Tink에 잘 맞는다

Tink의 하네스는 사용자가 명시적으로 호출하는 절차형 워크플로우에 가깝다. 이는 Matt이 선호한 방식과 일치한다.

좋은 procedure harness는 다음을 만족해야 한다.

- 언제 쓰는지 명확하다.
- 시작 전 필요한 입력을 정리한다.
- 에이전트가 해야 할 순서를 제공한다.
- 검증과 산출물 형식을 정한다.
- 사용자가 최종 판단을 잃지 않게 한다.

### 4. AFK 작업에는 샌드박스와 큐가 필요하다

Tink가 AFK 작업을 더 지원하려면 단순 명령보다 큐와 격리가 중요하다.

가능한 확장 방향:

- GitHub issue/label 기반 작업 큐
- `explore`, `implement`, `review`, `verify` 같은 단계 라벨
- 샌드박스 실행 정책
- PR 단위 결과 회수
- 실패 로그와 재시도 기준
- 자동 병합 가능 범위와 사람 리뷰 필요 범위 분리

### 5. 리뷰는 하네스 개선 루프의 핵심 데이터다

에이전트 결과를 사람이 리뷰할 때, 단순히 수정만 하고 끝내면 하네스는 좋아지지 않는다. 리뷰에서 다음 정보를 추출해야 한다.

- 어떤 지시가 누락됐는가
- 어떤 파일이나 문맥을 못 찾았는가
- 검증 명령이 충분했는가
- 실패가 코드 문제였는가, 하네스 문제였는가
- 다음 실행 전에 문서/스킬/테스트 중 무엇을 보강해야 하는가

이는 Tink의 `weave`나 memory 후보 기록과 직접 연결할 수 있다.

## 실천 체크리스트

- 현재 하네스와 스킬이 기본 컨텍스트를 얼마나 차지하는지 점검한다.
- 자동 호출이 필요한 ability와 사용자가 호출할 procedure를 분리한다.
- 반복 실패가 확인되지 않은 하네스는 추가하지 않는다.
- 각 하네스에 "완료 조건"과 "검증 증거"를 명시한다.
- AFK 작업은 loop보다 queue로 모델링한다.
- 리뷰 결과를 하네스 개선 후보로 남긴다.
- 코드베이스 구조 개선을 token optimization 전략으로 취급한다.
- 새 모델 평가보다, 기존 모델이 일하기 쉬운 저장소 구조와 테스트를 먼저 개선한다.

## 내 해석

이 영상은 "AI를 잘 쓰는 법"보다 "AI가 일할 수 있는 조직과 코드베이스를 어떻게 설계할 것인가"에 가깝다. 특히 Tink 관점에서는 하네스의 목적을 더 분명하게 만든다.

Tink가 추구해야 할 방향은 거대한 만능 프롬프트 묶음이 아니다. 관찰된 반복 문제를 작고 명확한 procedure로 만들고, 실행 결과를 검증하고, 리뷰에서 얻은 피드백을 다시 하네스에 반영하는 시스템이다.

즉, 좋은 Tink 하네스는 모델을 대신 똑똑하게 만드는 문서가 아니라, 사람이 전략적 판단을 유지한 채 에이전트에게 전술적 작업을 안전하게 위임하도록 돕는 작업 계약이어야 한다.
